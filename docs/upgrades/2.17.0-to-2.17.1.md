# Upgrade NKP from 2.17.0 to 2.17.1

> Upgrade path: NKP 2.17.0 → 2.17.1

This guide upgrades NKP applications and Kubernetes clusters on Nutanix AHV.

!!! warning "Required starting version"
    Start from NKP 2.17.0. Complete supported intermediate upgrades first when
    the environment runs an older release.

## Before you begin

You need administrative access, the NKP 2.17.1 CLI, a supported target node
image, current backups, healthy clusters, and capacity for rolling node
replacement.

## Upgrade order

1. Kommander on the management cluster.
2. Platform applications in each user workspace.
3. The management cluster Kubernetes version and node image.
4. Each managed workload cluster.

Attached clusters retain their external infrastructure lifecycle.

## Install the target CLI

```bash
tar -xvzf nkp-bundle_v2.17.1_linux_amd64.tar.gz
sudo install -m 0755 ./nkp-v2.17.1/cli/nkp /usr/local/bin/nkp
nkp version
```

## Verify the starting state

```bash
export MANAGEMENT_KUBECONFIG="$HOME/nkp-mgmt-cluster.conf"
export KUBECONFIG="${MANAGEMENT_KUBECONFIG}"

kubectl get nodes
kubectl get clusters -A
kubectl get machines -A
```

Resolve unhealthy nodes, failed Cluster API conditions, and degraded
applications before continuing.

## Upgrade Kommander

```bash
nkp upgrade kommander --kubeconfig "${MANAGEMENT_KUBECONFIG}"
kubectl get pods -A
```

## Upgrade workspace applications

```bash
nkp get workspaces --kubeconfig "${MANAGEMENT_KUBECONFIG}"

export WORKSPACE_NAME=dev-workspace
nkp upgrade workspace "${WORKSPACE_NAME}" \
  --kubeconfig "${MANAGEMENT_KUBECONFIG}"
```

Repeat for each user workspace and verify its applications.

## Prepare the target node image

Upload the supported image to Prism Central and record its exact name:

```bash
export TARGET_VM_IMAGE=nkp-rocky-9.7-release-cis-1.34.3.qcow2
```

The filename is an example. Use the image approved for the environment.

## Upgrade CLI-managed clusters

Upgrade the management cluster:

```bash
export MANAGEMENT_CLUSTER_NAME=nkp-mgmt-cluster

nkp upgrade cluster nutanix \
  --cluster-name "${MANAGEMENT_CLUSTER_NAME}" \
  --vm-image "${TARGET_VM_IMAGE}" \
  --kubeconfig "${MANAGEMENT_KUBECONFIG}"
```

After it is healthy, upgrade a workload cluster:

```bash
export WORKLOAD_CLUSTER_NAME=nkp-dev-cluster
export WORKLOAD_CLUSTER_NAMESPACE=dev-workspace

nkp upgrade cluster nutanix \
  --cluster-name "${WORKLOAD_CLUSTER_NAME}" \
  --namespace "${WORKLOAD_CLUSTER_NAMESPACE}" \
  --vm-image "${TARGET_VM_IMAGE}" \
  --kubeconfig "${MANAGEMENT_KUBECONFIG}"
```

## Upgrade GitOps-managed clusters

When Flux owns the cluster definition:

1. Update its declared Kubernetes version and node-image reference.
2. Review and merge the Git change.
3. Confirm successful Flux reconciliation.
4. Watch Cluster API perform the rolling replacement.

Do not combine imperative and GitOps ownership.

## Monitor and verify

```bash
kubectl get clusters,machines -A
kubectl get kubeadmcontrolplanes,machinedeployments -A

kubectl --kubeconfig "${MANAGEMENT_KUBECONFIG}" get nodes
kubectl --kubeconfig "$HOME/${WORKLOAD_CLUSTER_NAME}.conf" get nodes
kubectl --kubeconfig "$HOME/${WORKLOAD_CLUSTER_NAME}.conf" get pods -A
```

Confirm application replicas, storage, endpoints, monitoring, and backups.

When the complete fleet reports NKP 2.17.1 and is healthy, continue with
[Upgrade NKP from 2.17.1 to 2.18.0](2.17.1-to-2.18.0.md).
