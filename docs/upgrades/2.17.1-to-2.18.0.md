# Upgrade NKP from 2.17.1 to 2.18.0

> Upgrade path: NKP 2.17.1 → 2.18.0

This procedure separates the NKP platform upgrade from the Kubernetes 1.35 node
upgrade.

!!! warning "Required starting version"
    Complete [the 2.17.0 to 2.17.1 upgrade](2.17.0-to-2.17.1.md) first.
    Do not upgrade an NKP 2.17.0 environment directly to 2.18.0.

## Critical prerequisites

### Infrastructure versions

NKP 2.18.0 requires Prism Central 7.5 or later and AOS 7.5 or later. Older
Prism Central versions can return `404 Not Found` from the Nutanix v4 API during
preflight checks.

### Fleet health

Every management and managed workload cluster must be on NKP 2.17.1. Verify
backups, capacity for rolling replacement, disruption budgets, and Cluster API
health.

### Cluster names

Workload cluster names must not exceed 58 characters. Longer names can break
workspace application deployment, and Kubernetes clusters cannot be renamed in
place.

```bash
kubectl get clusters -A
```

### Legacy EKS clusters

NKP 2.18 removes legacy EKS commands and controllers. Migrate workloads from
non-ClusterClass EKS clusters and decommission those clusters before upgrading
Kommander.

### Istio

Record custom `IstioOperator` values, gateways, ambient-mode settings, and
namespaces with `istio-injection=enabled`. Custom Operator installations require
the manual migration described later.

## Required upgrade sequence

1. Upgrade Kommander, including `kommander-workspace`.
2. Upgrade the management cluster Kubernetes version and node image.
3. Upgrade applications in user workspaces.
4. Upgrade managed workload clusters.

Never upgrade a workload cluster before its management cluster. Attached
clusters retain their external Kubernetes lifecycle.

## Verify the starting state

```bash
export MANAGEMENT_KUBECONFIG="$HOME/nkp-mgmt-cluster.conf"
export KUBECONFIG="${MANAGEMENT_KUBECONFIG}"

kubectl get nodes
kubectl get clusters -A
kubectl get machines -A
kubectl get pods -A
```

## Install the NKP 2.18 CLI

```bash
tar -xvzf nkp-bundle_v2.18.0_linux_amd64.tar.gz
sudo install -m 0755 ./nkp-v2.18.0/cli/nkp /usr/local/bin/nkp
nkp version
```

Keep the 2.17.1 binary while the fleet contains both releases.

## Upgrade Kommander

```bash
nkp upgrade kommander --kubeconfig "${MANAGEMENT_KUBECONFIG}"
kubectl get pods -A
```

Do not separately upgrade `kommander-workspace`.

## Prepare the Kubernetes 1.35 image

Upload the supported image to Prism Central:

```bash
export TARGET_VM_IMAGE=nkp-rocky-9.7-release-cis-1.35.2.qcow2
```

The filename is an example. Use the patch version and operating system approved
for the environment.

## Discover the management namespace

An upgraded 2.17.1 management cluster normally remains in `default`; a new 2.18
cluster normally uses `kommander`. Discover rather than infer the namespace:

```bash
kubectl get clusters -A

export MANAGEMENT_CLUSTER_NAME=nkp-mgmt-cluster
export MANAGEMENT_CLUSTER_NAMESPACE=default
```

## Upgrade the management cluster

For a CLI-managed cluster:

```bash
nkp upgrade cluster nutanix \
  --cluster-name "${MANAGEMENT_CLUSTER_NAME}" \
  --namespace "${MANAGEMENT_CLUSTER_NAMESPACE}" \
  --vm-image "${TARGET_VM_IMAGE}" \
  --kubeconfig "${MANAGEMENT_KUBECONFIG}"
```

When Flux owns the cluster definition, update the declared Kubernetes version
and node image in Git instead. Do not combine imperative and GitOps ownership.

Wait for healthy nodes, Cluster API resources, and applications.

## Upgrade user workspaces

```bash
nkp get workspaces --kubeconfig "${MANAGEMENT_KUBECONFIG}"

export WORKSPACE_NAME=dev-workspace
nkp upgrade workspace "${WORKSPACE_NAME}" \
  --kubeconfig "${MANAGEMENT_KUBECONFIG}"
```

Repeat for every user workspace, but not `kommander-workspace`.

## Upgrade managed workload clusters

!!! danger "Check the known admission-webhook issue"
    An NKP 2.18.0 issue can reject a Kubernetes 1.34 → 1.35 update because of a
    ClusterClass version mismatch. Confirm that the installed build contains the
    fix or obtain the supported Nutanix workaround before proceeding.

For a CLI-managed cluster:

```bash
export WORKLOAD_CLUSTER_NAME=nkp-dev-cluster
export WORKLOAD_CLUSTER_NAMESPACE=dev-workspace

nkp upgrade cluster nutanix \
  --cluster-name "${WORKLOAD_CLUSTER_NAME}" \
  --namespace "${WORKLOAD_CLUSTER_NAMESPACE}" \
  --vm-image "${TARGET_VM_IMAGE}" \
  --kubeconfig "${MANAGEMENT_KUBECONFIG}"
```

For a Flux-managed cluster, update the version and node-image declaration in Git,
confirm reconciliation, and watch the rolling replacement. Upgrade clusters in
controlled batches.

## Validate cgroup v2 compatibility

Kubernetes 1.35 removes cgroup v1 support. Test applications and agents that
inspect cgroups directly, especially:

- Java 8 older than 8u372;
- Java 11 older than 11.0.16;
- monitoring or security agents that inspect cgroups;
- workloads deriving heap or thread settings from host resources.

Older runtimes can read total node resources instead of container limits,
causing `OOMKilled` containers or unexpected CPU behavior. Patch them before
moving the cluster to Kubernetes 1.35.

## Migrate Istio manually

The Operator deployment is not migrated automatically. The Helm-based Istio 1.29
replacement uses:

- `istio-helm-base`
- `istio-helm-cni`
- `istio-helm-istiod`
- `istio-helm-ztunnel`
- `istio-helm-gateway`

After these releases are healthy:

1. Translate custom `IstioOperator` values to supported Helm values.
2. Change namespace labels from `istio-injection=enabled` to
   `istio.io/rev=istio-helm`.
3. Roll application workloads during the maintenance window.
4. Verify gateways, sidecars or ambient components, policies, and connectivity.
5. Remove the legacy Operator only after all workloads have migrated.

## Prism Central Projects 2.0

This concerns Prism Central Projects, not NKP projects:

- upgraded clusters keep their existing user-defined Prism Central project;
- new NKP 2.18 clusters on Prism Central 7.6 must use `_internal`.

## Monitor and verify

```bash
kubectl get clusters,machines -A
kubectl get kubeadmcontrolplanes,machinedeployments -A

kubectl --kubeconfig "${MANAGEMENT_KUBECONFIG}" get nodes
kubectl --kubeconfig "${MANAGEMENT_KUBECONFIG}" get pods -A
kubectl --kubeconfig "$HOME/${WORKLOAD_CLUSTER_NAME}.conf" get nodes
kubectl --kubeconfig "$HOME/${WORKLOAD_CLUSTER_NAME}.conf" get pods -A
```

Confirm Cluster API readiness, application health, storage, external endpoints,
monitoring, backups, and Istio connectivity where applicable. Remove obsolete
images only after the complete fleet is stable.
